# 观察者模式与发布订阅模式的区别

> 有一种花，两种鸟，花定时开放，鸟看到花开会叫，鸟的叫声不一样，用js代码来实现这样一种场景

## 观察者模式

**主体（被观察者）**：定时开放的花

**观察者**：两种不同的鸟

**规则**：花开放 → 自动通知所有鸟 → 两只鸟发出不同叫声

```javascript
// ====================== 1. 被观察者（花）：负责发布“开花”事件 ======================
class Flower {
  constructor() {
    this.observers = []; // 存放所有观察者（鸟）
  }

  // 注册观察者（把鸟加入监听列表）
  addObserver(observer) {
    this.observers.push(observer);
  }

  // 通知所有观察者（开花了）
  notifyObservers() {
    console.log("🌸 花朵准时开放了！");
    this.observers.forEach((bird) => bird.cry()); // 让每只鸟叫
  }

  // 花定时开放（定时器模拟）
  openOnTime(seconds) {
    console.log(`⏳ 花朵将在 ${seconds} 秒后开放...`);
    setTimeout(() => {
      this.notifyObservers(); // 时间到 → 通知所有鸟
    }, seconds * 1000);
  }
}

// ====================== 2. 观察者（鸟）：收到开花通知就叫 ======================
// 鸟 A
class BirdA {
  cry() {
    console.log("🐦 鸟A：叽叽喳喳～叽叽喳喳！");
  }
}

// 鸟 B
class BirdB {
  cry() {
    console.log("🦜 鸟B：啾啾～啾啾啾！");
  }
}

// ====================== 3. 场景运行 ======================
// 1. 创建一朵花
const flower = new Flower();

// 2. 创建两只鸟
const birdA = new BirdA();
const birdB = new BirdB();

// 3. 把鸟注册到花的观察者列表
flower.addObserver(birdA);
flower.addObserver(birdB);

// 4. 启动：花 3 秒后定时开放
flower.openOnTime(3);
```

**被观察者（Flower）**

- 维护观察者列表
- 状态变化（开花）时**主动通知**所有观察者

**观察者（BirdA/BirdB）**

- 实现统一的响应方法（`cry()`）
- 收到通知后**执行自己的行为**

**解耦**

- 花不需要知道有多少只鸟、是什么鸟
- 鸟不需要主动轮询花是否开放
- 新增鸟、修改花的开放时间都非常方便

## 发布订阅模式

```javascript
// 1. 全局事件总线（发布订阅中心）
class EventBus {
  constructor() {
    // 存储事件：key=事件名，value=订阅者回调数组
    this.eventMap = {};
  }

  // 订阅：监听某个事件
  on(eventName, callback) {
    if (!this.eventMap[eventName]) {
      this.eventMap[eventName] = [];
    }
    this.eventMap[eventName].push(callback);
  }

  // 发布：触发某个事件
  emit(eventName, ...args) {
    if (!this.eventMap[eventName]) return;
    this.eventMap[eventName].forEach(cb => cb(...args));
  }

  // 取消订阅（可选）
  off(eventName, callback) {
    if (!this.eventMap[eventName]) return;
    this.eventMap[eventName] = this.eventMap[eventName].filter(cb => cb !== callback);
  }
}

// 2. 花：只负责定时发布「花开」事件，不关心有哪些鸟
class Flower {
  constructor(eventBus) {
    this.eventBus = eventBus;
  }

  // 定时开花，到点发布事件
  timerOpen(delay = 3000) {
    console.log(`⏰ 花朵倒计时 ${delay/1000} 秒后开放...`);
    setTimeout(() => {
      console.log("🌸 花开了，发布【花开通知】");
      // 发布事件
      this.eventBus.emit('flowerOpen');
    }, delay);
  }
}

// 3. 鸟1：订阅花开事件，自己做自己的叫声
class BirdOne {
  constructor(eventBus) {
    // 订阅事件
    eventBus.on('flowerOpen', () => {
      this.shout();
    });
  }

  shout() {
    console.log("🐦 第一只鸟：咕咕咕~~~");
  }
}

// 4. 鸟2：同样订阅花开事件，叫声不一样
class BirdTwo {
  constructor(eventBus) {
    eventBus.on('flowerOpen', () => {
      this.shout();
    });
  }

  shout() {
    console.log("🦜 第二只鸟：啾啾啾~~~");
  }
}

// ============ 场景使用 ============
// 实例化事件总线
const eventBus = new EventBus();

// 花、两只鸟都只依赖事件总线，互相完全不认识
const flower = new Flower(eventBus);
new BirdOne(eventBus);
new BirdTwo(eventBus);

// 启动定时开花
flower.timerOpen(3000);
```

**花不知道有几只鸟、是什么鸟**，只负责`emit`发布花开事件

**鸟不知道花什么时候开**，只负责`on`订阅事件

中间靠 `EventBus` 事件总线调度，**发布者和订阅者完全隔离**

后续再加第三只、第四只鸟，**不用改花的任何代码**，只新增订阅即可



## **发布订阅和观察者区别**

### 一、核心区别一句话总结

- **观察者模式**：观察者直接订阅目标对象（强耦合）
- **发布订阅模式**：通过中间媒介（事件中心）解耦（弱耦合）

------

### 二、结构对比

#### 1️⃣ 观察者模式（Observer）

结构很简单：

```
Subject（被观察者） <----> Observer（观察者）
```

特点：

- 被观察者**直接持有观察者列表**
- 状态变化时，主动通知所有观察者
- 观察者和被观察者**相互知道对方**

👉 典型代码（JS）：

```
class Subject {
  constructor() {
    this.observers = [];
  }

  subscribe(fn) {
    this.observers.push(fn);
  }

  notify(data) {
    this.observers.forEach(fn => fn(data));
  }
}
```

问题：

- 耦合高（Subject 知道 Observer）
- 不灵活（扩展困难）

------

#### 2️⃣ 发布订阅模式（Pub/Sub）

结构多了一层：

```
Publisher → EventBus → Subscriber
```

特点：

- 发布者不知道订阅者是谁
- 订阅者也不知道发布者是谁
- 中间靠“事件中心 / 消息队列”连接

👉 典型代码：

```
class EventBus {
  constructor() {
    this.events = {};
  }

  subscribe(event, fn) {
    (this.events[event] ||= []).push(fn);
  }

  publish(event, data) {
    (this.events[event] || []).forEach(fn => fn(data));
  }
}
```

优点：

- 完全解耦
- 可扩展（跨模块、跨系统）
- 更适合复杂系统

------

### 三、关键差异（重点）

| 维度         | 观察者模式 | 发布订阅模式     |
| ------------ | ---------- | ---------------- |
| 是否有中间层 | ❌ 没有     | ✅ 有（EventBus） |
| 耦合度       | 高         | 低               |
| 是否互相知道 | 知道       | 不知道           |
| 适用范围     | 单对象内   | 跨模块 / 系统    |
| 扩展性       | 一般       | 很强             |

------

### 四、你可以这样理解（很关键）

### 🧠 类比

- **观察者模式**
   👉 就像你关注了某个博主，他直接给你发消息
   （你和他有“直接关系”）
- **发布订阅模式**
   👉 你订阅了微博话题
   （博主发内容 → 微博平台 → 推给你）
   👉 你和博主根本不认识

------

### 五、前端中的实际例子

#### ✅ 观察者模式

- Vue 的响应式（早期实现更接近）
- DOM 的 MutationObserver
- 自定义对象监听

------

#### ✅ 发布订阅模式

- `EventEmitter`（Node.js）
- 前端 EventBus（Vue2 常见）
- Redux / Zustand（某种程度上）
- WebSocket 消息分发

------

### 六、什么时候用哪个？

#### 用观察者模式：

- 对象内部状态变化
- 简单一对多关系
- 不需要跨模块

#### 用发布订阅模式：

- 模块之间通信
- 大型应用
- 插件系统 / 微前端
- 异步事件流
